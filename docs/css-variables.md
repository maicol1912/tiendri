# CSS Variables Reference — Tiendri V2

> Auto-generated reference for all CSS custom properties output by `buildCssVars()`.
> Source: `src/lib/buildCssVars.ts`

---

## Overview

`buildCssVars(config: ResolvedStoreConfig): Record<string, string>` is the single
function that converts a fully resolved store config into a flat map of CSS custom
properties. The returned object is spread into a React `style` prop (cast to
`React.CSSProperties`) on the outermost layout wrapper.

### The 7 composable builders

| Builder function | Output prefix | Driven by |
|---|---|---|
| inline loop | `--t-{color-key}` | `config.colors` |
| `buildEffectVars` | `--t-fx-*`, `--t-shadow-scale` | `config.effects` + `config.layout.animationLevel` |
| `buildTypographyExtendedVars` | `--t-type-*` | `config.theme.typography` |
| `buildCardVars` | `--t-radius-*`, `--t-card-padding` | `config.layout.borderRadiusScale` + `config.layout.imageBorderRadius` |
| `buildLayoutVars` | `--t-grid-cols-*`, `--t-container-max`, `--t-image-fit` | `config.layout` |
| `buildColorVars` | `--t-card-bg-mode` | `config.theme.cardBackground` |
| inline (fonts) | `--font-body`, `--font-heading`, `--font-sans`, `--template-heading-font` | `config.font`, `config.headingFont` |

All builders fall back to `DEFAULT_PRESET_VALUES` (from `src/lib/presets/preset-defaults.ts`)
when a token is undefined — no variable is ever emitted with an `undefined` value.

## Color Variables

Generated by iterating `config.colors` (a `Partial<TemplateColorTokens>` merged object).
Every camelCase key is converted to kebab-case: `cardBg` → `--t-card-bg`.

```ts
// Conversion rule: camelCase → --t-{kebab-case}
"primary"       → --t-primary
"cardBg"        → --t-card-bg
"textMuted"     → --t-text-muted
"headerBg"      → --t-header-bg
```

## Typography Variables

Set when `config.theme?.typography` is present (i.e., after a preset has been applied).

### Heading variables (from `config.theme.typography`)

| Variable | Source field | Possible values | Default |
|---|---|---|---|
| `--t-type-heading-weight` | `typography.headingWeight` | `100`–`900` | `600` |
| `--t-type-heading-size` | `typography.headingScale` | `md→1.5rem`, `lg→2rem`, `xl→2.5rem`, `2xl→3.5rem` | `2rem` (lg) |
| `--t-type-heading-tracking` | `typography.headingTracking` | any CSS letter-spacing | `-0.01em` |
| `--t-type-heading-transform` | `typography.headingTransform` | `none`, `uppercase` | `none` |

### Extended typography variables (from `buildTypographyExtendedVars`)

| Variable | Source field | Possible values | Default |
|---|---|---|---|
| `--t-type-body-size` | `theme.bodyFontSize` | `sm→0.875rem`, `base→1rem`, `lg→1.125rem` | `1rem` |
| `--t-type-body-line-height` | `theme.bodyLineHeight` | `tight→1.35`, `normal→1.5`, `relaxed→1.65`, `loose→1.8` | `1.5` |
| `--t-type-body-weight` | hardcoded | `400` | `400` |
| `--t-type-display-size` | `theme.displaySize` | `md→2.5rem`, `lg→3.5rem`, `xl→5rem`, `2xl→7rem` | `3.5rem` |
| `--t-type-display-weight` | `typography.headingWeight` | `100`–`900` | `600` |
| `--t-type-display-tracking` | `typography.headingTracking` | any CSS letter-spacing | `-0.01em` |
| `--t-type-heading-style` | `theme.headingFontStyle` | `normal`, `italic` | `normal` |

| `--t-type-card-align` | `theme.cardTextAlign` | `left`, `center` | `left` |

**Consuming components:** product names, section headings, description paragraphs, hero

## Effect Variables

Generated by `buildEffectVars(effects)` where `effects` merges
`config.layout.animationLevel` / `config.layout.shadowStyle` with `config.effects.*`.

### Transition / animation

| Variable | Source field | Possible values | Default |
|---|---|---|---|
| `--t-fx-duration` | `effects.transitionSpeed` | `instant→0ms`, `fast→150ms`, `normal→250ms`, `slow→400ms`, `very-slow→600ms` | `250ms` |
| `--t-fx-easing` | `effects.transitionEasing` | `linear`, `ease`, `ease-in-out`, `spring→cubic-bezier(0.34,1.56,0.64,1)` | `ease` |

### Hover behavior (scale / lift / glow)

Magnitude is controlled by `effects.animationLevel`:

| Variable | `none` | `subtle` (default) | `full` |
|---|---|---|---|
| `--t-fx-hover-scale` | `1` | `1.02` | `1.05` |
| `--t-fx-hover-lift` | `0px` | `-2px` | `-4px` |
| `--t-fx-hover-glow-spread` | `0px` | `4px` | `12px` |
| `--t-fx-hover-glow-opacity` | `0` | `0.15` | `0.3` |

### Enter animation

| Variable | `none` | `subtle` (default) | `full` |
|---|---|---|---|
| `--t-fx-enter-distance` | `0px` | `8px` | `20px` |
| `--t-fx-enter-duration` | `0ms` | `300ms` | `500ms` |

### Shadow

| Variable | Source field | Possible values | Default |
|---|---|---|---|
| `--t-shadow-scale` | `effects.shadowElevation` | `none→0`, `xs→0.5`, `sm→1`, `md→1.5`, `lg→2`, `xl→3` | `1` (sm) |

**Consuming components:** `HOVER_EFFECT_MAP` and `IMAGE_HOVER_BEHAVIOR_MAP` in
`src/templates/_shared/style-maps.ts` use `--t-fx-duration`, `--t-fx-easing`,
`--t-fx-hover-scale`, `--t-fx-hover-lift`, `--t-fx-hover-glow-spread`,
`--t-fx-hover-glow-opacity`. Card layout components reference `--t-fx-duration` directly
for transition durations.

## Card Variables

Generated by `buildCardVars(cards, borderRadiusScale)`.

### Radius

| Variable | Source | Possible values | Default |
|---|---|---|---|
| `--t-radius-base` | `layout.borderRadiusScale` | `sharp→0`, `xs→0.125rem`, `sm→0.25rem`, `md→0.5rem`, `lg→0.75rem`, `xl→1rem`, `pill→9999px` | `0.5rem` (md) |
| `--t-radius-sm` | derived | `calc(var(--t-radius-base) * 0.5)` | `0.25rem` |
| `--t-radius-lg` | derived | `calc(var(--t-radius-base) * 2)` (or `9999px` when scale is `pill`) | `1rem` |
| `--t-radius-image` | `layout.imageBorderRadius` | `same-as-card→var(--t-radius-base)`, `none→0`, `rounded→0.5rem`, `circle→9999px` | `var(--t-radius-base)` |

> **Note:** Templates also define `config.radius` keys (e.g. `card`, `category`,
> `button`) that get emitted as `--t-radius-{key}`. These come from the template's own
> `config.ts`, not from `buildCardVars`. Example: `config.radius.button = "0px"` →
> `--t-radius-button: 0px`.

### Card padding and image

| Variable | Source field | Possible values | Default |
|---|---|---|---|
| `--t-card-padding` | `layout.cardPadding` (via `DEFAULT_PRESET_VALUES.layout`) | `none→0`, `tight→0.5rem`, `normal→0.75rem`, `spacious→1.25rem` | `0.75rem` (normal) |
| `--t-image-fit` | `cards.imageFit` | `cover`, `contain` | `cover` |

> `--t-image-fit` is emitted by `buildLayoutVars`, not `buildCardVars`, but belongs
> conceptually with card imagery.

**Consuming components:** card containers, image wrappers, button border-radius,
category chip border-radius.

Generated by `buildLayoutVars(layout)`.

| `--t-grid-cols-mobile` | `layout.gridColumnsMobile` | `1`, `2` | `2` |

> `grid-cols-${config.grid.products.mobile}` directly in component code — the CSS

## Color Application Variables

Generated by `buildColorVars(color)`.

| Variable | Source field | Possible values | Default |
|---|---|---|---|
| `--t-card-bg-mode` | `theme.cardBackground` | `white`, `surface`, `transparent`, `primary-tint` | `white` |

This token is a **semantic mode identifier**, not a color value. Components read it and
map it to the actual color token:

```ts
// Example: card background resolution
// "white"       → background: var(--t-card-bg)      (pure white)
// "surface"     → background: var(--t-surface)       (tinted surface)
// "transparent" → background: transparent
// "primary-tint"→ background: rgba(var(--t-primary-rgb), 0.05)
```

## Font Variables

Set directly from `config.font` (body) and `config.headingFont` (display).

| Variable | Source | Purpose |
|---|---|---|
| `--font-body` | `config.font` | Next.js font CSS variable for body font |
| `--font-sans` | `config.font` (mirrors body) | Alias used by fashion template components (`style={{ fontFamily: "var(--font-sans)" }}`) |
| `--font-heading` | `config.headingFont` | Next.js font CSS variable for heading font |
| `--template-heading-font` | `config.headingFont` | Raw `font-family` string for inline heading styles |

**Example values:**
```ts
config.font        = "Inter"
config.headingFont = "var(--font-display), ui-sans-serif, system-ui, sans-serif"
```

**Consuming components:** `Contained.tsx` (`var(--font-sans)`), any component with
inline `fontFamily` style, any element styled with `font-[var(--font-heading)]`.

## How to Add a New CSS Variable

### Step 1 — Add the token type

If the new value requires a new enum/union type, add it to
`src/types/templates/primitives.ts`:

```ts
export type MyNewToken = "option-a" | "option-b" | "option-c";
```

### Step 2 — Add the token to the right preset interface

In `src/lib/presets/preset-types.ts`, add the field to the relevant interface
(`TypographyTokens`, `LayoutTokens`, `CardTokens`, `EffectTokens`, or `ColorTokens`):

```ts
export interface LayoutTokens {
  // ... existing fields
  myNewToken?: MyNewToken;
}
```

### Step 3 — Add the default value

In `src/lib/presets/preset-defaults.ts`, add the default under the matching group:

```ts
export const DEFAULT_PRESET_VALUES = {
  layout: {
    // ... existing
    myNewToken: "option-a",
  },
  // ...
};
```

### Step 4 — Add a lookup map in `buildCssVars.ts`

In `src/lib/buildCssVars.ts`, add a `Record<MyNewToken, string>` map and wire it
into the appropriate builder function:

```ts
const MY_NEW_TOKEN_MAP: Record<MyNewToken, string> = {
  "option-a": "value-a",
  "option-b": "value-b",
  "option-c": "value-c",
};

function buildLayoutVars(layout: LayoutTokens): Record<string, string> {
  // ... existing
  const myNewToken: MyNewToken = layout.myNewToken ?? (defaults.myNewToken as MyNewToken);

  return {
    // ... existing
    "--t-my-new-token": MY_NEW_TOKEN_MAP[myNewToken],
  };
}
```

### Step 5 — Expose via StoreCustomization (if merchant-configurable)

If merchants should be able to change this from the dashboard, add the field to
`ThemeCustomization` or `LayoutCustomization` in
`src/types/templates/store-customization.ts`, and update
`src/lib/resolveTemplateConfig.ts` to forward it onto `ResolvedStoreConfig`.

### Step 6 — Use in components

Reference the new variable in Tailwind or inline styles:

```tsx
// Tailwind arbitrary value
<div className="border-[var(--t-my-new-token)]" />

// Inline style
<div style={{ borderRadius: "var(--t-my-new-token)" }} />
```

## Variable Budget

### Current count

| Group | Count |
|---|---|
| Color (`--t-{color}` from `config.colors`) | ~35 per template (varies by palette) |
| Color derived (`--t-primary-rgb`) | 1 |
| Typography (`--t-type-*`) | 9 |
| Spacing (`--t-space-*`) | 4 |
| Effect (`--t-fx-*` + `--t-shadow-scale`) | 9 |
| Card / radius (`--t-radius-*`, `--t-card-padding`) | 5 |
| Layout (`--t-grid-cols-*`, `--t-container-max`, `--t-image-fit`) | 4 |
| Color application (`--t-card-bg-mode`) | 1 |
| Fonts (`--font-*`, `--template-heading-font`) | 4 |
| **Total** | **~72** |

### Performance notes

- All variables are set on a single root wrapper element via `style={}` — one reflow,
  zero cascade traversal cost.
- Every variable value is a resolved CSS primitive (px, rem, ms, raw string). No
  `calc()` chains with unknown values at parse time except `--t-radius-sm` and
  `--t-radius-lg` which derive from `--t-radius-base`.
- Aim to keep total variable count under 100. Beyond that, consider grouping logical
  tokens into shorthand values or moving to Tailwind config-driven classes.
- Grid columns (`--t-grid-cols-mobile`, `--t-grid-cols-desktop`) are numbers, not
  class strings. Components construct `grid-cols-${n}` Tailwind classes at render time;
  the CSS variable version exists for contexts where the class cannot be pre-scanned by
  the JIT.
